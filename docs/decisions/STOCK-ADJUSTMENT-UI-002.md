# STOCK-ADJUSTMENT-UI-002 库存修改入口与界面规则

- 状态：已确认
- 确认日期：2026-07-12
- 适用范围：人工库存调整 Phase C 员工端

## 1. 入口位置

库存修改申请入口必须放在员工“库存管理”页面内。

- 员工从开单首页进入“库存管理”；
- 在库存管理页面内点击“申请修改库存”进入修改模式；
- 不在开单首页的顶层功能区单独增加“库存调整申请”入口；
- 管理端审核入口保持不变。

## 2. 视觉与交互

库存修改界面必须与现有开单页、库存管理页保持一致：

- 复用 `store-style.css` 的页面容器、返回按钮、品牌标签、规格标签、商品卡片、数量控件和底部提交按钮；
- 不使用独立且风格割裂的 `stock-adjustment.css` 作为员工端主视觉；
- 商品排序、品牌切换、规格切换和商品展示沿用现有开单/库存管理逻辑；
- 页面标题、按钮和提示文案使用中文。

## 3. 数量修改方式

库存修改类似开单页，但员工只修改“散数”。

- 每个商品显示当前库存；
- 每个商品只提供散数调整控件；
- 不提供箱数、盒数输入；
- 调整方向分为“增加”和“减少”，或使用带正负号的散数，但界面必须清楚且不容易误操作；
- 散数必须为非负整数，最终提交数量转换为带符号的最小库存单位；
- 允许调整后库存为负数；
- 修改已驳回或已撤回申请时，必须正确恢复原调整方向和散数，不得把减少误变成增加。

## 4. 页面流程

建议在 `store_stock.html` / `store-app.js` 中实现两种状态：

1. 查看库存：只展示当前库存；
2. 申请修改：显示与开单页一致的商品列表及散数调整控件。

申请修改状态应提供：

- 调整原因；
- “其他”原因说明；
- 备注；
- 已选择商品与预计库存；
- 保存并提交审核；
- 我的待审核/已驳回申请；
- 历史申请入口。

## 5. 验收标准

- 开单首页不再单独显示“库存调整申请”；
- 库存管理页面内能明显进入修改申请；
- 员工端视觉与开单、库存管理页面一致；
- 只能输入整数散数；
- 不出现箱、盒输入；
- 增加/减少方向编辑和重提时保持正确；
- 员工端不再直接更新 `van_stocks`，仍通过申请与审核 RPC 完成正式库存变化。
